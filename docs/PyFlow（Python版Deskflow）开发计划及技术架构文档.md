# PyFlow（Python版Deskflow）开发计划及技术架构文档

# 一、文档概述

## 1.1 文档目的

本文档明确 PyFlow 项目（Python 版 Deskflow 键鼠共享工具）的技术架构设计、开发阶段规划、核心技术选型及实施细节，为开发过程提供标准化指导，确保项目贴合 Deskflow 核心逻辑，同时适配 Python 开发特性，实现“局域网内键鼠共享、剪贴板同步、屏幕边缘无缝切换”的核心目标。

## 1.2 项目定位

PyFlow 是一款基于 Python 开发的局域网键鼠共享工具，对标 Deskflow 官方功能，采用 C/S 架构，实现“一套键鼠控制多台局域网内电脑”，兼顾易用性与学习性，用于学习 Deskflow 核心原理，同时提供可直接使用的轻量化版本（非工业级，侧重学习与演示）。

## 1.3 核心目标

- 功能目标：实现鼠标共享、键盘共享、剪贴板文字同步、屏幕边缘切换（核心功能对标 Deskflow）；
- 性能目标：局域网内延迟控制在 10-50ms，无明显卡顿、飘移，支持每秒 100+ 键鼠事件传输；
- 学习目标：架构与 Deskflow 1:1 对应，代码模块化、易读，便于理解 C/S 架构、自定义协议、跨平台输入输出等核心技术；
- 跨平台目标：支持 Windows、macOS、Linux 三大系统（基础适配，优先保证 Windows/macOS 流畅运行）。

# 二、技术架构设计

## 2.1 整体架构（对标 Deskflow 官方 C/S 架构）

PyFlow 采用 **客户端/服务端（C/S）架构**，核心分为 5 大模块，与 Deskflow 官方架构完全对应，各模块职责清晰、低耦合，便于开发、调试与扩展。整体架构如下：

### 2.1.1 架构分层（从顶层到底层）

1. 应用层：负责程序入口、模式切换（服务端/客户端）、配置加载；
2. 业务层：负责核心业务逻辑（输入监听、输出模拟、边缘切换、剪贴板同步）；
3. 协议层：负责自定义二进制协议的打包与解析，实现低延迟数据传输；
4. 网络层：负责 TCP 长连接建立、数据收发、连接管理，可选 TLS 加密；
5. 底层适配层：负责调用系统原生接口，实现跨平台的输入监听与输出模拟。

### 2.1.2 核心模块详解

各模块职责、技术选型、对应 Deskflow 模块如下表所示：

| PyFlow 模块         | 核心职责                                                    | 技术选型（Python）                                  | 对应 Deskflow（C++）模块                   |
| ------------------- | ----------------------------------------------------------- | --------------------------------------------------- | ------------------------------------------ |
| 应用层（入口/配置） | 程序启动、模式选择（服务端/客户端）、配置加载（IP、端口等） | argparse（命令行参数）、configparser（配置文件）    | Main 入口、Config 模块                     |
| 服务端业务层        | 键鼠输入监听、屏幕边缘判断、事件分发                        | pynput（监听键鼠）、pyautogui（屏幕尺寸获取）       | InputMonitor、ScreenEdge 模块              |
| 客户端业务层        | 接收网络数据、解析事件、模拟键鼠/剪贴板输出                 | pynput（模拟键鼠）、pyperclip（剪贴板操作）         | OutputSimulator、Clipboard 模块            |
| 协议层              | 自定义二进制协议设计、事件打包、数据解析                    | struct（二进制打包/解析）                           | Protocol 模块                              |
| 网络层              | TCP 长连接、数据收发、连接管理、可选 TLS 加密               | socket（原生TCP）、ssl（加密）、threading（多线程） | Network 模块                               |
| 底层适配层          | 跨平台接口适配（Windows/macOS/Linux 输入输出接口）          | pynput（跨平台封装）、platform（系统判断）          | 系统原生接口封装模块（SendInput、IOHID等） |

## 2.2 核心技术细节

### 2.2.1 网络通信设计

- 通信协议：TCP 长连接（全程保持连接，避免频繁握手导致延迟）；
- 端口配置：默认端口 12345（可通过配置文件修改）；
- 加密支持：可选 TLS 加密（基于 Python ssl 模块，自签名证书，适配局域网场景）；
- 连接管理：服务端支持单客户端连接（简化开发，聚焦核心功能），客户端自动重连（避免连接断开）。

### 2.2.2 自定义二进制协议设计（对标 Deskflow 极简协议）

协议核心目标：极简、低延迟，单个数据包体积控制在 4-16 字节，适配键鼠事件高频传输需求，协议格式如下（3种核心数据包）：

1. 鼠标移动事件包（8字节）
   - 字节0：事件类型（0x01 = 鼠标移动）；
   - 字节1-4：X坐标（int32，大端序）；
   - 字节5-8：Y坐标（int32，大端序）。
2. 键盘事件包（6字节）
   - 字节0：事件类型（0x02 = 键盘事件）；
   - 字节1-2：按键码（int16）；
   - 字节3：按键状态（0x00 = 松开，0x01 = 按下）；
   - 字节4-5：保留位（备用）。
3. 剪贴板同步包（可变长度，最小4字节）
   - 字节0：事件类型（0x03 = 剪贴板同步）；
   - 字节1-2：文本长度（int16）；
   - 字节3+：UTF-8 编码的文本内容（长度对应字节1-2）。

### 2.2.3 跨平台适配设计

- Windows：基于 pynput 调用 Windows API（SendInput），支持鼠标、键盘全功能；
- macOS：基于 pynput 调用 IOHID 接口，需授权 accessibility 权限，支持核心功能；
- Linux：基于 pynput 调用 uinput 接口，需安装 libei、libportal 依赖（对标 Deskflow Linux 依赖），支持基础功能。

### 2.2.4 数据流设计（核心流程）

#### 服务端（主控电脑）数据流：

1. 启动服务端，监听指定端口，等待客户端连接；
2. 输入监听模块实时捕获鼠标、键盘事件；
3. 边缘检测模块判断鼠标是否滑到屏幕边缘（触发切换逻辑，暂不实现多客户端切换，聚焦单客户端）；
4. 协议模块将键鼠事件打包成二进制数据包；
5. 网络模块通过 TCP 长连接将数据包发送至客户端；
6. 剪贴板模块实时监测剪贴板变化，同步至客户端。

#### 客户端（被控电脑）数据流：

1. 启动客户端，连接服务端 IP 和端口；
2. 网络模块接收服务端发送的二进制数据包；
3. 协议模块解析数据包，识别事件类型（鼠标/键盘/剪贴板）；
4. 输出模拟模块根据解析结果，模拟鼠标移动、键盘按键或剪贴板同步；
5. 持续监听服务端数据，异常断开时自动重连。

## 2.3 文件架构设计（可直接用于开发）

> ⚠️ **提示**：本节为设计文档，描述的是项目初始规划的理想结构。实际实现中，部分文件命名和位置可能有所不同。当前实际结构请参考 [PyFlow开发计划.md](./PyFlow开发计划.md) 中的"项目文件结构"章节。（最后更新：2026-04-14）

严格遵循模块化设计，文件结构清晰，与技术架构一一对应，便于学习和维护，具体结构如下：

```plain
pyflow/
├── main.py                # 程序入口（启动服务端/客户端、加载配置）
├── config.py              # 配置文件（IP、端口、屏幕尺寸、加密开关等）
├── requirements.txt       # 依赖包清单（pynput、pyperclip等）
├── server/                # 服务端模块（主控电脑）
│   ├── __init__.py
│   ├── server_core.py     # 服务端主逻辑（连接管理、事件分发）
│   ├── input_monitor.py   # 键鼠输入监听（核心：捕获鼠标/键盘事件）
│   ├── screen_edge.py     # 屏幕边缘检测（判断是否触发切换）
│   └── clipboard_monitor.py # 剪贴板监听（同步至客户端）
├── client/                # 客户端模块（被控电脑）
│   ├── __init__.py
│   ├── client_core.py     # 客户端主逻辑（连接服务端、接收数据）
│   ├── output_simulator.py # 输出模拟（模拟鼠标/键盘/剪贴板）
│   └── reconnect.py       # 自动重连逻辑
├── network/               # 网络通用模块
│   ├── __init__.py
│   ├── tcp_server.py      # TCP服务端实现（监听、收发数据）
│   ├── tcp_client.py      # TCP客户端实现（连接、收发数据）
│   ├── protocol.py        # 自定义协议（打包/解析数据包）
│   └── tls_encrypt.py     # TLS加密（可选，自签名证书）
└── common/                # 通用工具模块
    ├── __init__.py
    ├── system_utils.py    # 系统判断、屏幕尺寸获取等通用工具
    └── logger.py          # 日志工具（调试、错误记录）
```

## 2.4 技术选型总结

所有技术选型均遵循“轻量化、易学习、贴合 Deskflow 逻辑”的原则，避免复杂框架，聚焦核心原理，具体选型如下：

- 核心语言：Python 3.8+（兼容性强、语法简洁，适合入门学习）；
- 网络通信：socket（原生TCP）、ssl（加密）；
- 键鼠操作：pynput（跨平台、轻量，支持监听与模拟）；
- 剪贴板操作：pyperclip（跨平台，支持文本同步）；
- 配置管理：configparser（简单易⽤，支持配置文件）；
- 日志工具：logging（Python原生，便于调试）；
- 二进制处理：struct（原生模块，打包/解析二进制数据）。

# 三、开发计划（分阶段实施，可落地、可验证）

开发周期：总计 4 周（20个工作日），分 4 个阶段，每个阶段有明确目标、任务和验收标准，确保循序渐进，聚焦核心功能，同时兼顾学习性。

## 3.1 阶段一：环境搭建与基础模块开发（第 1 周）

### 3.1.1 阶段目标

完成开发环境搭建、项目初始化、核心工具模块开发，确保基础功能可运行。

### 3.1.2 核心任务

1. 环境搭建：
   - 安装 Python 3.8+，配置虚拟环境；
   - 安装依赖包，编写 requirements.txt；
   - 搭建代码管理（如 Git），初始化项目结构。
2. 项目初始化：
   - 创建文件架构（按 2.3 节设计）；
   - 编写 config.py，定义默认配置（IP、端口等）；
   - 编写 common 模块（系统工具、日志工具）。
3. 基础模块开发：
   - network/tcp_server.py：实现 TCP 服务端基础监听、连接功能；
   - network/tcp_client.py：实现 TCP 客户端基础连接、收发功能；
   - common/system_utils.py：实现系统判断、屏幕尺寸获取功能。

### 3.1.3 验收标准

- 项目结构完整，所有文件夹、基础文件创建完成；
- TCP 服务端可正常监听端口，客户端可成功连接；
- 系统工具可正确识别当前操作系统，获取屏幕尺寸；
- 日志工具可正常记录调试信息、错误信息。

## 3.2 阶段二：核心协议与基础功能开发（第 2 周）

### 3.2.1 阶段目标

完成自定义二进制协议开发、服务端输入监听、客户端输出模拟，实现基础鼠标共享功能。

### 3.2.2 核心任务

1. 协议模块开发（network/protocol.py）：
   - 实现鼠标移动、键盘事件、剪贴板同步的二进制打包与解析；
   - 编写协议测试代码，验证数据包打包/解析正确性。
2. 服务端输入监听开发（server/input_monitor.py）：
   - 使用 pynput 监听鼠标移动、点击事件；
   - 将监听的事件传递给协议模块，打包成二进制数据。
3. 客户端输出模拟开发（client/output_simulator.py）：
   - 使用 pynput 模拟鼠标移动、点击事件；
   - 接收协议解析后的事件，执行对应模拟操作。
4. 整合测试：
   - 实现服务端发送鼠标事件、客户端接收并模拟移动；
   - 调试延迟，确保局域网内鼠标移动无明显卡顿。

### 3.2.3 验收标准

- 协议模块可正确打包/解析鼠标、键盘、剪贴板事件；
- 服务端可正常监听鼠标事件，打包后发送至客户端；
- 客户端可接收数据，正确解析并模拟鼠标移动；
- 局域网内鼠标移动延迟 ≤ 50ms，无明显飘移。

## 3.3 阶段三：完整功能开发与优化（第 3 周）

### 3.3.1 阶段目标

完成键盘共享、剪贴板同步、屏幕边缘检测功能，优化延迟与稳定性，实现跨平台基础适配。

### 3.3.2 核心任务

1. 键盘共享功能开发：
   - 完善 input_monitor.py，监听键盘按键（按下/松开）；
   - 完善 output_simulator.py，模拟键盘按键；
   - 整合测试，确保键盘按键可同步至被控电脑。
2. 剪贴板同步功能开发：
   - 编写 server/clipboard_monitor.py，监听剪贴板变化；
   - 完善 client/output_simulator.py，实现剪贴板同步；
   - 测试文本剪贴板同步，确保双向同步（可选，优先实现服务端→客户端）。
3. 屏幕边缘检测开发（server/screen_edge.py）：
   - 判断鼠标是否滑到屏幕边缘（如右侧边缘）；
   - 实现基础切换逻辑（暂不支持多客户端，仅做边缘判断提示）。
4. 优化与适配：
   - 优化协议解析速度，减少延迟；
   - 添加客户端自动重连功能（client/reconnect.py）；
   - 适配 Windows、macOS 系统，解决权限问题（如 macOS accessibility 授权）。

### 3.3.3 验收标准

- 键盘按键可同步（按下/松开），无漏键、错键；
- 剪贴板文本可同步，同步延迟 ≤ 100ms；
- 屏幕边缘检测准确，鼠标滑到边缘可触发提示；
- 客户端断开连接后可自动重连，稳定性良好；
- Windows、macOS 系统可正常运行核心功能。

## 3.4 阶段四：功能测试、文档完善与优化（第 4 周）

### 3.4.1 阶段目标

完成全面测试、修复 bug，完善开发文档与使用文档，优化用户体验，确保项目可正常使用、便于学习。

### 3.4.2 核心任务

1. 全面测试：
   - 功能测试：验证鼠标、键盘、剪贴板、边缘检测所有功能；
   - 兼容性测试：在 Windows 10+/macOS 12+/Linux（Ubuntu 22.04）测试运行；
   - 压力测试：长时间运行（2小时以上），检查是否卡顿、断开连接；
   - bug 修复：解决测试中发现的延迟、漏事件、重连失败等问题。
2. 文档完善：
   - 完善技术架构文档（补充代码注释、模块调用关系）；
   - 编写使用文档（如何启动服务端/客户端、配置修改、常见问题）；
   - 编写代码注释，确保代码易读、便于学习。
3. 最终优化：
   - 优化延迟，尽可能将鼠标延迟控制在 30ms 以内；
   - 优化日志输出，便于调试；
   - 添加启动提示、错误提示，提升用户体验。

### 3.4.3 验收标准

- 所有核心功能正常运行，无重大 bug；
- 三大系统（Windows/macOS/Linux）均可正常适配；
- 文档完整（技术文档、使用文档），代码注释清晰；
- 长时间运行稳定，无卡顿、断开连接问题；
- 项目可直接运行，新手可根据文档快速上手。

# 四、风险与应对措施

# 五、总结

PyFlow 项目以 Deskflow 为对标，采用 C/S 架构，通过 Python 实现键鼠共享、剪贴板同步等核心功能，整体架构清晰、模块化强，既适合学习 Deskflow 核心原理，也能提供可直接使用的轻量化工具。

开发计划分 4 个阶段循序渐进，从基础环境搭建到完整功能开发、测试优化，每个阶段有明确目标和验收标准，可确保项目有序推进。在开发过程中，重点关注延迟优化、跨平台适配和代码可读性，兼顾学习性与实用性。
